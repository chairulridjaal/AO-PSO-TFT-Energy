# Hybrid AO-PSO Optimised Temporal Fusion Transformer for Smart Grid Multi-Horizon Energy Forecasting

> **A two-phase meta-heuristic cascade that unifies Aquila Optimizer global exploration with Particle Swarm local exploitation for automated TFT hyperparameter tuning, validated under statistical rigor across heterogeneous smart grid datasets.**

---

## 1. Abstract

Temporal Fusion Transformers (TFT) [3] deliver state-of-the-art multi-horizon accuracy for energy load forecasting, yet their performance is acutely sensitive to hyperparameter selection -- a challenge compounded by the mixed continuous-discrete nature of the configuration space and the non-trivial architectural constraint $d_{\text{model}} \bmod n_{\text{heads}} = 0$. This work proposes a **Hybrid Aquila Optimizer -- Particle Swarm Optimization (AO-PSO)** algorithm that decomposes the tuning problem into two complementary phases: (1) Aquila Optimizer global exploration via a four-phase hunting strategy [1] to identify the basin of attraction containing the global optimum, followed by a deterministic **hand-off** to (2) PSO velocity-position refinement [2] for precise convergence within that basin. The approach is validated on two heterogeneous energy datasets -- a micro-grid household (UCI [11], France) and a macro-grid regional aggregate (ISO-NE [10], USA) -- both enriched with population-weighted meteorological features from the Open-Meteo API [5] backed by ERA5 [6], ERA5-Land [7], and CERRA [8] reanalyses, across four forecasting horizons (1h, 6h, 12h, 24h), with statistical significance established via 10-run Wilcoxon signed-rank tests [4] ($\alpha = 0.05$).

## 3. The Hybrid AO-PSO Algorithm

### 3.1 Motivation

Neither standalone optimizer is sufficient alone:

- **AO** excels at **global exploration** via Levy flights and spiral trajectories, but converges slowly in the final refinement stage.
- **PSO** excels at **local exploitation** via velocity-position updates, but is susceptible to premature convergence when initialised randomly.

The hybrid eliminates both weaknesses by **chaining** them: AO maps the global landscape, then PSO fine-tunes within the discovered basin.

### 3.2 Two-Phase Cascade

```
Phase 1: GLOBAL MAP (Aquila Optimizer)
═══════════════════════════════════════════════════════════════
   Iterations 1 → ⌊ao_fraction × T⌋

   The AO population executes four hunting phases:
     • Expanded Exploration   (high-altitude soaring)
     • Narrowed Exploration   (contour flight with Lévy spiral)
     • Expanded Exploitation  (slow controlled descent)
     • Narrowed Exploitation  (walk-and-grab with QF decay)

   Phase selection governed by progress ratio t/T and random r:
     t/T ≤ 2/3 ∧ r ≤ 0.5  →  Expanded Exploration
     t/T ≤ 2/3 ∧ r > 0.5  →  Narrowed Exploration
     t/T > 2/3 ∧ r ≤ 0.5  →  Expanded Exploitation
     t/T > 2/3 ∧ r > 0.5  →  Narrowed Exploitation

                    ┌──────────────────────────────────┐
                    │        T H E   H A N D - O F F   │
                    ├──────────────────────────────────┤
                    │  PSO receives:                   │
                    │   • AO's final population        │
                    │   • AO's personal-best fitnesses │
                    │   • AO's global-best position    │
                    │   • Velocities initialised to 0  │
                    └──────────────────────────────────┘

Phase 2: LOCAL FINE-TUNING (Particle Swarm Optimization)
═══════════════════════════════════════════════════════════════
   Iterations ⌊ao_fraction × T⌋ + 1 → T

   Standard PSO velocity-position update with linearly
   decaying inertia weight w ∈ [w_min, w_max]:

     v_i(t+1) = w·v_i(t) + c₁·r₁·(pBest_i − x_i) + c₂·r₂·(gBest − x_i)
     x_i(t+1) = x_i(t) + v_i(t+1)
```

### 3.3 Search Space

The optimizers tune five TFT hyperparameters with a post-update constraint enforcement:

| Dim | Parameter            | Type       | Range                  | Notes                                                         |
| --- | -------------------- | ---------- | ---------------------- | ------------------------------------------------------------- |
| 0   | `learning_rate`      | Continuous | $[10^{-5},\; 10^{-2}]$ | Optimised in $\log_{10}$ space                                |
| 1   | `d_model`            | Discrete   | $\{32, 64, 128, 256\}$ | Snapped to nearest allowed value                              |
| 2   | `num_encoder_layers` | Discrete   | $\{1, 2, 3, 4\}$       | Transformer stack depth                                       |
| 3   | `n_heads`            | Discrete   | $\{1, 2, 4, 8\}$       | **Constraint:** $d_{\text{model}} \bmod n_{\text{heads}} = 0$ |
| 4   | `dropout`            | Continuous | $[0.05,\; 0.50]$       | Applied throughout the TFT                                    |

The divisibility constraint is enforced after **every** position update via `_enforce_constraints()`: if $d_{\text{model}} \bmod n_{\text{heads}} \neq 0$, the nearest valid $n_{\text{heads}}$ that divides $d_{\text{model}}$ is selected.

### 3.4 Proxy Fitness Evaluation

Each candidate hyperparameter configuration is evaluated cheaply via a **proxy training** protocol:

1. Instantiate `BaseTFT` with the candidate's decoded hyperparameters.
2. Train for `proxy_epochs` (default: 2) on a `subset_fraction` (default: 30%) of training batches.
3. Evaluate full validation RMSE as the fitness score to minimise.

This provides an informative but inexpensive signal at ~1/50th the cost of a full training run. The stochastic nature of both AO and PSO is inherently robust to this noise [1, Section 4.2].

## 4. Anti-Leakage Data Protocol

All data handling in `dataset_utils.py` enforces a strict temporal integrity protocol:

- **Chronological splitting only** -- train/val/test partitions respect temporal order; no random shuffling across time boundaries.
- **Scaler fitted exclusively on training data** -- `StandardScaler` statistics ($\mu$, $\sigma$) are computed from the training set only. Validation and test sets are transformed using these train-fitted parameters.
- **Controlled shuffling** -- the training DataLoader shuffles samples within its window-safe range; validation and test DataLoaders iterate in strict temporal order.
- **Sliding window isolation** -- the `TimeSeriesDataset` constructs non-overlapping forecast targets, preventing any future information from leaking into input windows.

### Group A: Optimizer Convergence Speed

| Optimizer         | Implementation        | Strategy                                        |
| ----------------- | --------------------- | ----------------------------------------------- |
| Pure AO           | `AquilaOptimizer` [1] | 4-phase hunting (explore → exploit)             |

**Metrics:** Iterations to convergence, optimisation wall-time (seconds), total GPU compute time.

### Group B: Forecasting Accuracy

Compares the champion AO-PSO-TFT against established baselines on held-out test data.

| Model          | Class                      | Category                             |
| -------------- | -------------------------- | ------------------------------------ |
| LSTM           | `StandardLSTM`             | Deep learning sequence baseline      |
| Bi-LSTM        | `BiLSTM`                   | Bidirectional deep learning baseline |
| XGBoost        | `XGBoostBaseline`          | Gradient-boosted tree baseline       |
| Untuned TFT    | `BaseTFT` (default params) | Proves optimisation necessity        |
| **AO-PSO-TFT** | `BaseTFT` (hybrid-tuned)   | **Hybrid-optimised champion**        |

**Metrics:** RMSE, MAE, MAPE evaluated at four horizons ($h \in \{1, 6, 12, 24\}$ hours) on both micro-grid [11] and macro-grid [10] datasets.

**Evaluation Metrics [9] (real-world units, post inverse-scaling):**

$$\text{RMSE} = \sqrt{\frac{1}{N}\sum_{i=1}^{N}(y_i - \hat{y}_i)^2}$$

$$\text{MAE} = \frac{1}{N}\sum_{i=1}^{N}|y_i - \hat{y}_i|$$

$$\text{MAPE} = \frac{100}{N}\sum_{i=1}^{N}\frac{|y_i - \hat{y}_i|}{|y_i| + \epsilon}$$

## 6. Statistical Rigor: The 10x Grind Protocol

Every model configuration is executed **10 independent times** with different random seeds ($s \in \{1, 2, \ldots, 10\}$). Results are reported as $\bar{x} \pm \sigma$ and statistical significance is established via the **Wilcoxon signed-rank test** [4]:

$$H_0: \text{median}(e_{\text{Champion}} - e_{\text{Baseline}}) = 0$$

$$H_1: \text{median}(e_{\text{Champion}} - e_{\text{Baseline}}) \neq 0$$

A $p$-value below $\alpha = 0.05$ rejects $H_0$, confirming that the observed performance difference is statistically significant and not attributable to random seed variation.

## 8. Running the Experiments

### 8.1 Headless Execution via `run_experiments.py`

The full experiment pipeline is consolidated into a **headless, crash-resilient CLI application** designed for unattended execution on remote GPU servers (tested on EPYC / RTX 3090 instances via Vast.ai).

```bash
# Full pipeline: baselines → standalone swarms → hybrid champion
python run_experiments.py --all

# Individual phases
python run_experiments.py --baselines     # Phase 1: LSTM, BiLSTM, XGBoost (NB01)
python run_experiments.py --swarms        # Phase 2: Pure AO, Pure PSO (NB02)
python run_experiments.py --hybrid        # Phase 3: Hybrid AO-PSO (NB03)

# Compose phases
python run_experiments.py --baselines --hybrid
```

### 8.2 Crash Resilience & Checkpointing

Results are saved to disk after **every single completed run**. If the process crashes, SSH drops, or the GPU instance is preempted:

1. Re-launch the **same command**.
2. The script detects completed runs in the JSON checkpoint files and **skips them automatically**.
3. At worst, only the single in-flight run at crash time is lost.

All output is dual-logged to both `stdout` and `results/execution_10x_grind.log`.

### 8.3 SSH-Resilient Execution with tmux

```bash
# Create a named session
tmux new -s grind

# Launch the pipeline
python run_experiments.py --all

# Detach: Ctrl+B, then D
# SSH can now safely disconnect.

# Reconnect later:
tmux attach -t grind
```
